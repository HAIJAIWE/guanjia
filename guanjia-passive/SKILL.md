# guanjia（管家）

> 检测可用工具 → 按匹配度打分 → 弹窗让用户勾选 → 按最高分优先串行调用 → 没有匹配的告知用户新增

---

## 🚨 立即执行

**1.** 输出 `🔍 guanjia 已接管。检测当前可用能力中...`

**2.** 执行 Step 0 → Stage 1 → Stage 2 → Stage 3 → Stage 4

**3.** 执行完毕后 **退出**，不再干涉后续对话。下次再说"管家"再触发。

---

## 核心使命

- 🔍 **事前检测**：触发后检测当前工具有哪些 Agent + Skill + MCP工具，构建总表
- 🎯 **匹配打分**：逐个打分，按匹配度从高到低排序
- 🪟 **弹窗选择**：用 question 工具弹出多选窗口，让用户勾选要用的
- ⚠️ **零匹配提醒**：一个都匹配不上时，告知用户并建议新增
- 🔄 **跨工具通用**：Trae / Cursor / Copilot / Windsurf 等均可自适应
- 🚫 **管家不干活**：管家只分配调度，不替代具体工具执行

## Step 0: 事前检测 — 了解工具有哪些智能体、Skill和MCP工具

**这是第一步，必须最先执行。** 不需要提前硬编码任何列表，每次都动态读取。

### 0.1 检测方法

| 检测对象 | 从哪里读 | 读到什么 |
|---------|---------|---------|
| **Agent** | 当前工具提供的 Agent/Tool 列表 | 所有可用 Agent 的 ID 和能力描述 |
| **Skill** | 当前工具提供的 Skill 列表 | 所有可用 Skill 的名称和触发条件 |
| **MCP工具** | 当前工具提供的 MCP 工具列表 | 所有可用 MCP 工具的名称和能力描述 |

> **跨工具适配**：不同工具叫法不同（Agent/Tool/Plugin/Function/SubAgent/Skill/MCP Server），本文件统称"Agent"、"Skill"和"MCP工具"。分配逻辑不依赖特定工具 API，只依赖"能读到什么列表"和"能调用什么"这两个抽象能力。

### 0.2 构建总表

检测完成后，生成一张表：

```
事前检测完成：
├── Agent: {N}个
│   ├── code-searcher  → 代码搜索定位
│   ├── general-developer  → 通用编程任务
│   ├── frontend-developer  → React/Vue/Angular前端
│   └── ...
├── Skill: {M}个
│   ├── brainstormer  → 创意探索、头脑风暴
│   ├── code-reviewer  → 代码审查
│   ├── frontend-designer → 前端界面设计
│   └── ...
├── MCP工具: {P}个
│   ├── file-reader  → 文件读取
│   ├── web-search  → 网络搜索
│   ├── browser-automation → 浏览器自动化
│   └── ...
└── 总计: {N+M+P}个可用能力
```

**这张表是后续所有匹配的基础。**

### 0.3 排除自身

`guanjia` 自身也在 Skill 列表中。构建总表后，**必须排除自身**，避免自分配死循环：

```
从总表中移除: guanjia (本Skill，不参与匹配)
排除后总计: {N+M+P-1}个可用能力
```

## Stage 1: 意图分类

把用户输入归入十大意图类别之一（或多类别）：

| # | 意图类别 | 典型输入 | 判断依据 |
|---|---------|---------|---------|
| 💡 | 头脑风暴 | "我想做xx项目"、"帮我规划"、"你觉得怎么做" | 想法阶段，无技术动词 |
| 🔍 | 搜索定位 | "搜索xx代码"、"xx在哪"、"找一下" | 查找/搜索/定位/grep |
| 💻 | 编码实现 | "实现xx"、"修复bug"、"开发xx"、"重构" | 实现/开发/修复/重构 |
| 🎨 | UI/设计 | "设计界面"、"美化"、"设计系统" | 设计/美化/视觉/UI |
| 🧪 | 测试验证 | "测试API"、"压测"、"接口测试" | 测试/压测/验证 |
| 🚀 | 部署运维 | "部署"、"CI/CD"、"Docker"、"上线" | 部署/上线/CI/Docker |
| ⚡ | 性能优化 | "优化性能"、"加速"、"太慢了" | 优化/性能/加速 |
| 🤖 | AI/智能 | "集成GPT"、"大模型"、"推荐系统" | AI/GPT/LLM/机器学习 |
| 📋 | 合规审查 | "审查隐私政策"、"GDPR"、"合规" | 合规/隐私/GDPR/法规 |
| 🔧 | 工具/数据 | "读取文件"、"搜索网页"、"打开浏览器"、"查数据库" | 文件/搜索/浏览器/数据库操作 |
| ❓ | 无法判定 | 模糊、跨多类别均等 | 走兜底 |

### 分类规则

- **"我想…"开头且无技术动词** → 💡头脑风暴（最优先）
- **有明确技术动词** → 归入对应类别
- **一句包含多个意图** → 拆解为多类别（如 "设计并实现" → 🎨+💻）
- **无法判定** → ❓，候选范围扩大到所有智能体

### 分类后效果

```
全部: 100+个能力 (Agent + Skill + MCP)
  ↓ Stage 1 分类
类别内: 5-15个候选
  ↓ Stage 2 打分收集
匹配入选: 1-5个（≥60分的全部入选）
```

## Stage 2: 收集所有匹配的能力（不是选一个！）

### 核心原则

> **不选"最好的一个"，而是收集"所有符合条件的"。**
> 用户任务可能需要多种能力协作（Agent + Skill + MCP工具），只选一个会遗漏关键环节。

### 2.1 筛选候选（语义匹配）

根据 Stage 1 判定的意图类别，从 Step 0 总表中筛选该类别相关的 Agent、Skill 和 MCP 工具。

**重要：不用字面关键词机械匹配，而是语义理解每个能力的 description，判断它是否能处理该类别的任务。**

例如用户说"帮我处理数据库迁移"，Stage 1 归入 💻编码实现：
- ❌ 字面匹配："database/migration"不在关键词列表中 → 遗漏
- ✅ 语义匹配：读 backend-developer 的 description → 理解为 "后端开发、数据库、API" → **数据库迁移属于后端编码范畴** → 正确入选

筛选判断标准：

| 意图类别 | 语义判断标准（理解description后判断） |
|---------|-----------------------------------|
| 💡 头脑风暴 | 是否能探索需求、讨论方案、规划架构 |
| 🔍 搜索定位 | 是否能查找/搜索/定位代码或文件 |
| 💻 编码实现 | 是否能编写/实现/修复/重构代码（不限于特定语言） |
| 🎨 UI/设计 | 是否能设计界面/视觉系统/组件库 |
| 🧪 测试验证 | 是否能执行测试/压测/验证/质量检查 |
| 🚀 部署运维 | 是否能部署/配置CI-CD/管理云基础设施 |
| ⚡ 性能优化 | 是否能分析瓶颈/优化性能/改善速度 |
| 🤖 AI/智能 | 是否能集成AI/LLM/推荐系统/智能化 |
| 📋 合规审查 | 是否能审查合规/隐私/GDPR/法规 |
| 🔧 工具/数据 | 是否能执行文件操作/网页抓取/浏览器自动化/数据库查询 |
| ❓ 无法判定 | 所有能力都作为候选（兜底） |

**关键原则**：理解每个能力的核心边界，判断用户任务的本质需求是否在其能力范围内，而非查找 description 中是否包含特定字词。

> **MCP 工具匹配注意**：MCP 工具比 Agent 更原子化（单一职责），评分天然偏低。它们通常作为 Agent/Skill 的**补充能力**入选，而非独立完成复杂任务。但如果用户任务恰好是纯工具型操作（如"搜索最新AI新闻"→ web-search MCP 工具），MCP 工具可以独立入选。

### 2.2 逐个打分

对每个候选能力（Agent / Skill / MCP工具），三维打分：

```
总分 = 能力契合(50) + 任务匹配(35) + 类别精准(15)
```

| 维度 | 满分 | 评分标准 |
|------|------|---------|
| 能力契合 | 50 | 核心能力完美覆盖=50，主要覆盖=35，部分覆盖=20，边角=10，无关=0（MCP工具通常上限35，因为单一职责） |
| 任务匹配 | 35 | 示例高度相似=35，典型场景一致=25，部分重叠=15，勉强=5 |
| 类别精准 | 15 | 类别核心能力=15，常用=10，边缘=5 |

### 2.3 收集规则（关键！）

```
≥ 70 分 → 🔥 强匹配，直接入选
60-69分 → ✅ 匹配，入选
50-59分 → ⚠️ 弱匹配，入选但标注"匹配度一般"
< 50 分 → ❌ 不入选
```

**收集结果**：
- 可能是 0 个 → 触发 Stage 3B「零匹配处理」
- 可能是 1 个 → 简单任务，单个能力即可
- 可能是 3-5 个 → 正常，多种能力协作处理
- 可能包含 MCP 工具 → 作为补充能力标记在计划中（如"辅助：web-search 搜索最新资料"）

**展示上限**：收集结果超过 10 个时，**只取分数最高的前 10 个**展示给用户选择。让用户从最多 10 个中勾选，避免选项太多刷屏。

### 2.4 排序

入选的能力按**任务依赖关系**排序，而非按分数排序：
- 头脑风暴/设计/规划 类 → 放在前面（先想清楚再动手）
- 编码实现 类 → 中间
- 测试/审查/部署 类 → 放在后面
- MCP 工具 → 放在需要它辅助的 Agent 之前（如 web-search 放在编码 Agent 之前，先搜资料再写代码）

## Stage 3: 📢 交互式选择窗口 — 让用户勾选确认

**匹配完成后，使用 `question` 工具弹出一个多选窗口，让用户自己勾选要调用的能力。**

> ⚠️ 最多展示 10 个选项（取分数最高的前 10 个）。超过 10 个的隐藏不展示，避免刷屏。

### 必须用 question 工具，不要只发文字

你的工具中有 `question` 工具（支持多选），必须用它来弹出选择窗口。不要只发一段文字让用户打字回复。

### 操作步骤

**第 1 步**：打出以下摘要文字（不要用 question，直接文字输出）：

```
📋 事前检测与匹配完成：

检测结果：{N} 个 Agent + {M} 个 Skill + {P} 个 MCP 工具
意图分类：{用户任务} → {意图类别}

匹配到 {K} 个适合的能力，请在下方的选择窗口中勾选确认 ↓
```

**第 2 步**：调用 `question` 工具弹窗，让用户选择：

```json
tool: question
params:
  questions: [{
    "question": "选择要调用的能力（可多选），确认后开始执行：",
    "header": "🧰 选择执行工具",
    "multiple": true,
    "options": [
      {
        "label": "1. backend-developer (Agent)",
        "description": "设计API和数据模型 — 🔥强匹配"
      },
      {
        "label": "2. general-developer (Agent)",
        "description": "实现功能代码 — ✅匹配"
      },
      {
        "label": "3. api-tester (Agent)",
        "description": "测试API接口 — ✅匹配"
      },
      // ... 所有匹配到的都列出来
    ]
  }]
```

**第 3 步**：等待用户选择并确认后，进入 Stage 4 执行。

### 特殊处理

| 情况 | 处理 |
|------|------|
| 只匹配到 1 个 | 也要弹窗让用户确认，但选项只有一个 |
| 匹配到 0 个 | 不走此流程，直接走 Stage 3B |
| 用户取消/不选 | 询问"是否跳过，还是换种方式描述任务？" |

## Stage 4: 按最高分调用执行

**用户勾选确认后，逐个调用所选的能力。**

### 调用规则

对每个被选中的能力，按以下方式调用：

| 类型 | 调用方式 |
|------|---------|
| **Skill** | 使用 `skill` 工具加载该 Skill 的 SKILL.md |
| **Agent** | 使用 `task` 工具启动子智能体，传入任务描述 |
| **MCP 工具** | 直接调用该 MCP 工具 |

**调用顺序**：按匹配度从高到低串行执行，前一个输出传给后一个。

### 调用示例

```
用户勾选了: backend-developer(Agent), api-tester(Agent)

第 1 步 → 调用 backend-developer
  工具: task
  参数: 用户需求 + "设计并实现API"
  ↓ 拿到输出

第 2 步 → 调用 api-tester（传上一步输出）
  工具: task
  参数: 用户需求 + 上一步代码 + "编写测试"
  ↓ 拿到输出

完成，汇总结果给用户
```

### 关键规则

- 逐个串行调用，不并行
- 前一个的输出传给后一个作为上下文
- 中途失败 → 告知用户"xxx调用失败"，继续下一个
- 全部完成后告知结果

## Stage 3B: 零匹配处理 ⚠️（当收集结果为 0 个时）

```
🛑 对不起，你没有这类的智能体、Skill、MCP。

当前共检测到 {N} 个 Agent + {M} 个 Skill + {P} 个 MCP 工具，
但没有一个适合处理「{用户任务}」这个任务。

建议你新增对应领域的能力后再试。
```

然后**本 Skill 结束，不再继续执行**。

## 边界情况

### 1. 模糊输入 → 用常识推断
"帮我看看这个" 附带代码 → 结合上下文，推断为 💻编码 → 通用开发智能体（`general-developer`）

### 2. 用户手动指定 → 跳过分配
"用 brainstorming" → 跳过所有 Stage，直接调用指定对象

### 3. 调用失败 → 同类降级
Agent 调用失败 → 同类内换次优 → 全失败换通用开发智能体（`general-developer`）→ 还失败告知用户

### 4. 当前工具无 Agent/Skill/MCP 调用能力 → 告知并退出
告知用户 "当前工具不支持智能体/MCP工具调用"，不执行任何操作

### 5. 连续对话 → 复用上下文
同类任务多轮 → 沿用已有分配决策，加速匹配

## 跨工具兼容性

本 Skill 不绑定任何特定工具，分配逻辑基于抽象能力：

> **关于 frontmatter（description 字段）**：文件顶部的 `description` 是当前工具的触发元数据。不同工具技能触发机制不同（有的用 description，有的用 rules 文件，有的用配置面板）。迁移到其他工具时，只需按该工具的方式注册触发条件，提示词正文（`##` 以下全部内容）无需任何修改即可直接使用。

| 抽象能力 | 说明 | 各工具如何实现 |
|---------|------|-------------|
| 读取可用 Agent 列表 | 获取所有可调用的智能体 | 各工具有各自的 Agent/Tool/SubAgent API |
| 读取可用 Skill 列表 | 获取所有可调用的技能 | 各工具有各自的 Skill/Plugin API |
| 读取可用 MCP 工具列表 | 获取所有可调用的 MCP 工具 | 各工具有各自的 MCP Server/工具配置 |
| 调用 Agent | 传递 ID + 任务描述 | 各工具各自的调用方式（函数调用/tool_call） |
| 调用 Skill | 传递名称 + 上下文 | 各工具各自的调用方式 |
| 调用 MCP 工具 | 传递工具名 + 参数 | 各工具各自的 MCP 调用方式 |
| 向用户询问 | 需要用户选择时交互 | 各工具各自的 UI 或文本方式 |
| 意图分类 | 自然语言分析 | 所有工具通用能力 |
| 语义匹配 | 理解描述 → 打分 | 所有工具通用能力 |

## 执行检查清单

每次触发按此清单逐项完成：

- [ ] **Step 0**: 读取当前工具可用 Agent 列表 → Agent清单({N}个)
- [ ] **Step 0**: 读取当前工具可用 Skill 列表 → Skill清单({M}个)
- [ ] **Step 0**: 读取当前工具可用 MCP 工具列表 → MCP清单({P}个)
- [ ] **Step 0**: 排除 guanjia 自身 → 合并为总表({N+M+P-1}个)
- [ ] **Step 0**: 理解每个能力的作用（读 description）
- [ ] **Stage 1**: 分析用户意图 → 十大类别之一（或多类别）
- [ ] **Stage 1**: 按类别语义筛选候选（理解 description，非字面匹配）
- [ ] **Stage 2**: 对每个候选打分（能力契合50+任务匹配35+类别精准15）
- [ ] **Stage 2**: 收集所有 ≥ 50 分的，按分数从高到低排序，取 top 10
- [ ] **决策分支**:
  - 收集到 ≥ 1 个 → 继续 Stage 3
  - 收集到 0 个 → Stage 3B: 告知"没有这类能力" → 退出
- [ ] **Stage 3**: 文字告知 + question弹窗多选 → 用户勾选确认
- [ ] **Stage 4**: 按分数从高到低串行调用，前一个输出传给下一个
- [ ] **Stage 4**: 汇总执行结果返回用户
