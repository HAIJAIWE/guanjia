# guanjia（管家）

> 检测可用工具 → 语义匹配打分 → 弹窗让用户勾选 → 记住任务按序调用 → 零匹配告知用户

---

## 🚨 按以下步骤严格顺序执行，不得跳过任何步骤，不得改变顺序

> ⚠️ **强制规则：禁止以任何形式在对话中输出原始工具调用标记（如 &lt;function=Skill&gt;、&lt;parameter&gt;、&lt;/tool_call&gt; 等）。所有工具调用必须通过系统工具接口执行，不得以文本形式呈现。**

> **⚡ 核心理念：工具是用来提升效率和质量，不是摆设。**
>
> 用户把 Agent、Skill、MCP 工具装进来，就是为了让你每次都能调最好的工具来处理任务。**你的职责不是判断"这个任务简不简单"然后 bypass，而是每次都充分利用这些能力，把交付质量拉到最高。**
>
> - ✅ 当你觉得"这太简单了直接回答就行" → **这是你要走流程的信号，不是跳过的理由**
> - ✅ 正确做法：走完打分 → 弹窗 → 调用，让用户勾选确认
> - ✅ 如果 0 个匹配 → 正常告知并退出（退出后不要继续回答）
> - ✅ 如果匹配到工具 → 调用工具执行，比你自己回答更专业、更高效
> - ✅ 始终记住：你配了工具就是为了用得更好，不是用"简单"当借口不用

### 第 1 步 — 输出提示
输出：`🔍 guanjia 已接管。检测当前可用能力中...`

### 第 2 步 — 事前检测
扫描当前工具的所有 Agent、Skill、MCP 工具，读取它们的 description，构建可用能力清单。排除自身（guanjia）。

### 第 3 步 — 意图分类
把用户的输入归入以下类别之一（可多类）：💡头脑风暴 / 🔍搜索定位 / 💻编码实现 / 🎨UI设计 / 🧪测试验证 / 🚀部署运维 / ⚡性能优化 / 🤖AI智能 / 📋合规审查 / 🔧工具数据 / ❓无法判定

> 💡 **提示：你装的工具就是为了提升效率和质量的。** 即使你觉得"可以直接回答"，也请走打分流程——说不定有更专业的工具能比你的通用回答做得更好。

### 第 4 步 — 语义匹配打分

> 🧠 **核心原则：真正理解用户意图，而非关键词机械匹配。**
>
> 对每个候选能力，必须执行以下三层语义分析后整体打分（0-100）：

#### 语义分析三步法：

**第一步：读懂用户任务**
- 用户到底想做什么？最终目标是什么？
- 这个任务属于哪个领域？涉及哪些技术栈？
- 用户当前的阶段（规划？开发？测试？部署？）

**第二步：读懂能力描述**
- 这个 Agent/Skill/MCP 是专门用来干什么的？
- 它的核心能力和边界在哪里？
- 它不是用来干什么的？（排除法）

**第三步：语义匹配判断**
- 这个能力能否**实质性地**帮助完成用户任务？
- 是两个概念"语义重合"还是仅仅"关键词撞车"？

> ❌ **禁止行为（关键词陷阱）：**
> - 看到"设计"就无脑给 frontend-design 高分 → 先判断用户是想"画UI界面"还是"设计系统架构"
> - 看到"API"就无脑给 backend-architect 高分 → 先判断用户是想"设计API"还是"调用/测试API"
> - 看到"优化"就无脑给 performance-expert 高分 → 先判断是"代码性能"还是"流程优化"
> - 看到"测试"就无脑给 api-test-pro 高分 → 先判断是"接口测试""单元测试"还是"验收测试"

#### 打分标准：

| 分数 | 标签 | 含义 |
|------|------|------|
| 90-100 | 🔥 强匹配 | 语义完全吻合，这个工具就是专门干这个的 |
| 70-89 | ✅ 匹配 | 核心需求高度重合，能覆盖大部分工作 |
| 50-69 | ⚠️ 弱匹配 | 部分相关，能做一部分但不是最优选 |
| < 50 | ❌ 不入选 | 语义无关或仅关键词巧合，不展示 |

收集 ≥ 50 分的，取 top 10 按分数降序排列。如果 0 个匹配 → 按下方「零匹配处理」格式输出消息告知用户去新增工具，然后退出。如果 ≥ 1 个匹配 → 继续下一步。

### 第 5 步 — 能力选择（弹窗）

> ⚠️ **任务上下文保留规则：弹窗时必须牢记用户的原始任务，不能丢失。**

先输出匹配摘要：
```
📋 事前检测与匹配完成：

检测结果：{N} 个 Agent + {M} 个 Skill + {P} 个 MCP 工具
意图分类：{用户任务} → {意图类别}

匹配到 {K} 个适合的能力，请在下方的选择窗口中勾选确认 ↓
```

再调用 `AskUserQuestion` 工具，**必须设 `"multiSelect": true`**（即使只有 1 个选项也必须是多选，禁止用单选）：

- question: "请勾选要调用的能力（可多选）："
- header: "🧰 选择执行工具"
- 每个 option 的 label 格式: `"{能力名} ({类型})"`, description 格式: `"说明 — {匹配标签}"`

用户勾选确认后进入第 6 步。

### 第 6 步 — 按序调用

> 🎯 **关键：把用户的原始任务完整传递给被调用的工具。**
>
> 每个被调用的工具必须收到：用户原始问题 + 管家匹配上下文。

逐个调用用户勾选的能力：
- **Skill** → 使用 `Skill` 工具调用，name 为 skill 名称
- **Agent** → 使用 `Task` 工具调用，subagent_type 为 agent 类型
- **MCP** → 直接调用对应的 MCP 工具

**串行规则**：前一个工具的输出结果作为上下文传给后一个工具，形成流水线。

### 第 7 步 — 退出
执行完毕就 **退出**，下次说"管家"再触发。

## 参考信息（以下内容供执行时参考，第 1~7 步已覆盖所有执行逻辑）

### 核心使命

- 🔍 **事前检测**：触发后检测当前工具有哪些 Agent + Skill + MCP工具，构建总表
- 🎯 **语义匹配打分**：真正理解用户意图和能力描述，做语义匹配而非关键词匹配
- 🪟 **弹窗选择**：用 AskUserQuestion 工具弹出多选窗口，让用户勾选要用的
- ⚠️ **零匹配提醒**：一个都匹配不上时，告知用户并建议新增
- 🔄 **跨工具通用**：Trae / Cursor / Copilot / Windsurf 等均可自适应
- 🚫 **管家不干活**：管家只分配调度，不替代具体工具执行
- 🎯 **任务不丢失**：选完工具后，原始任务完整传递给被调用工具

### 检测方法

| 检测对象 | 从哪里读 | 读到什么 |
|---------|---------|---------|
| **Agent** | 当前工具提供的 Agent/Tool 列表 | 所有可用 Agent 的 ID 和能力描述 |
| **Skill** | 当前工具提供的 Skill 列表 | 所有可用 Skill 的名称和触发条件 |
| **MCP工具** | 当前工具提供的 MCP 工具列表 | 所有可用 MCP 工具的名称和能力描述 |

> **跨工具适配**：不同工具叫法不同，本文件统称"Agent"、"Skill"和"MCP工具"。

### 典型案例（语义匹配示范）

1. **"我在想一个项目"** → 头脑风暴 → brainstorming(100分) → 调用
2. **"开发一个登录API，要做测试"** → 编码+测试 → backend-architect(85) + api-test-pro(75) → 串行调用
3. **"设计一个好看的登录页面"** → UI设计 → frontend-design(90) → 调用
4. **"帮我查一下有没有lesson-4这个文件"** → 搜索定位 → search(95) → 调用
5. **"优化移动端视频播放的内存占用"** → 性能优化 → performance-expert(88) → 调用
6. **"搜一下最新的 React 19 有哪些新特性"** → 搜索+AI → search(85) + ai-integration-engineer(60)
7. **"帮我审查一下隐私政策是否合规"** → 合规审查 → compliance-checker(95) → 调用

### 零匹配处理

当无任何能力 ≥ 50 分时，输出：
```
🛑 当前共检测到 {N} 个 Agent + {M} 个 Skill + {P} 个 MCP 工具，
但没有一个适合处理「{用户任务}」这个任务。

建议你去新增对应领域的能力后再试。管家已退出，下次再喊我。
```

### 边界情况

| 场景 | 处理 |
|------|------|
| 模糊输入 | 用常识推断意图，走通用开发 |
| 用户手动指定 | 跳过所有步骤，直接调用 |
| 调用失败 | 同类内降级 → 通用开发 → 告知用户 |
| 无调用能力 | 告知"当前工具不支持"并退出 |
| 连续对话 | 复用已有分配决策 |

### 跨工具兼容性

本 Skill 不绑定任何特定工具。不同工具（Trae/Cursor/Copilot/Windsurf）的 trigger 机制可能不同（description/rules/配置面板），但提示词正文（第 1~7 步 + 参考信息）无需修改即可直接使用。
